# AI assistant session model

## Summary

The AI assistant session model defines server-side persistence for in-app AI conversation history, run metadata, tool events, and compacted context. It lets users keep multiple assistant sessions per workspace, switch between them, and continue a conversation without sending the whole historical transcript on every provider call.

Session storage is host-local application state, not canonical master data. It is scoped to the resolved workspace that owns the `masterdata/` directory and is stored below `os.UserConfigDir()` so chat history does not pollute exported project data or Git-managed YAML files.

For MasterDataMate-managed chat sessions, the persisted prompt context represents what the host is willing to send to an API-level model provider on later turns. When context compaction runs, older message content in the prompt context may be replaced by a summary. The recommended implementation keeps enough revision metadata to explain that compaction happened, while treating the latest compacted prompt context as the source of truth for future managed assistant runs.

Delegated external agents such as Codex CLI do not use this model as their internal prompt memory. MasterDataMate may keep a display session and task records for UI continuity, but the external agent owns its own session state, compaction, and tool loop.

## Storage Location

The host stores sessions under a workspace-scoped directory:

```text
<os.UserConfigDir()>/
  masterdatamate/
    <workspaceBaseName>-<workspacePathHash>/
      ai-sessions/
        sessions.json
        <sessionId>.json
```

`workspaceBaseName` is the base name of the resolved workspace root, which is the parent directory that contains `masterdata/`. `workspacePathHash` is a deterministic lowercase hash of the cleaned absolute workspace root path. The hash must include the absolute path so two projects with the same directory name do not collide.

The storage root is derived only by the backend. Browser clients and AI providers receive session IDs and display names, not host filesystem paths.

## Rules / Constraints

- Session logs are stored by the server-side host. The browser may cache UI state for responsiveness, but the host is authoritative after reload.
- Session storage must be outside `masterdata/`, outside generation folders, and outside exported artifacts.
- The host must create the workspace session directory with owner-only permissions when the platform supports them.
- Session IDs, run IDs, and message IDs are opaque. Clients must not derive filesystem paths from them.
- Session file writes must be atomic: write a temporary file in the same directory, flush when practical, then rename into place.
- A corrupted session file must not prevent the application from opening the workspace. The host reports a diagnostic and leaves the session unavailable until repaired or deleted.
- The AI provider receives only the selected session's budgeted prompt context and scoped current UI/workspace context. Other sessions are never sent implicitly.
- This session model owns prompt-context compaction only for `managed_chat_agent` sessions. `delegated_external_agent` sessions rely on the external agent's own memory and compaction behavior.
- Raw API keys, credential references that can resolve secrets, provider request headers containing secrets, and OS keyring diagnostics must never be written to session files.
- Raw absolute workspace paths should not be placed in prompt messages unless a specific tool result requires them and the user-facing workflow has made path exposure clear. Session metadata may keep hashed workspace identity only.
- Tool-call previews for side-effecting actions may be retained in session history, but confirmation tokens are short-lived runtime objects and must not be reusable after reload.
- Archived sessions remain readable and can be resumed by explicitly unarchiving or selecting them. Deleted sessions are hidden from normal lists.

## Context Budgeting And Compaction

- Each run builds a provider request from system instructions, the selected session's current prompt-context messages, and the current scoped UI/workspace context.
- The host must estimate token cost before a run using provider token counting when available, or a conservative local estimate otherwise.
- The service should reserve output tokens and tool-result headroom instead of filling the provider window with history.
- Recent user and assistant turns should be preserved verbatim where possible. Older turns, repeated diagnostics, large tool results, and stale UI context are the first compaction candidates.
- Compaction produces a `summary` message that preserves durable user intent, unresolved decisions, accepted constraints, pending proposals, important tool results, and references to affected tables, generations, and schemas.
- Compaction may rewrite the `messages` array because it is the prompt context for future provider calls. The `compactions` array records which messages were replaced so the UI can explain why older detail is summarized.
- If a future implementation needs stronger auditability, it may add an append-only raw event log beside the prompt-context file. The prompt context remains the data sent to providers, while the raw log is never required for ordinary chat continuation.
- Users should be able to trigger manual compaction for a long session and should see when automatic compaction occurred.
- Compaction itself must use the active managed chat provider only after applying the same privacy and tool restrictions as ordinary assistant runs. A local deterministic summarizer or local provider is preferred when available for sensitive workspaces, but the system must also support hosted providers because local LLMs are not always available.
- Automatic compaction may run through the same hosted provider selected for the managed chat session when no local summarizer is available. The UI should expose that compaction will send the older prompt context to that provider under the same data-handling assumptions as an ordinary assistant run.
- Delegated external agent sessions do not run MasterDataMate prompt compaction. For Codex CLI, the host delegates context window management, summarization, and continuation behavior to Codex.

## Session Operations

- Create session: creates a new active session with no user messages and selects it in the AI panel.
- List sessions: returns active and archived session metadata for the current workspace, ordered by `updated_at` descending by default.
- Load session: returns one session's visible messages, run summaries, retained tool cards, and compaction markers.
- Rename session: updates the title without modifying messages.
- Archive session: hides the session from the default active list while keeping it resumable.
- Delete session: removes or tombstones the session.
- Switch session: changes the selected session without discarding unsaved table, schema, generation, or plugin edits.
- Compact session: for `managed_chat_agent`, rewrites old prompt-context messages into a summary and appends a compaction record. For `delegated_external_agent`, this action is hidden or mapped only to an external-agent-supported cleanup command if one exists.

## Best-Practice Guidance

- Store two concepts separately when possible: the UI timeline users read and the provider prompt context the model receives. The first may retain rich cards and diagnostics; the second should be budgeted and compact.
- Prefer loss-aware compaction over unbounded transcript replay. Sending every old message degrades latency, cost, and model focus, while carefully summarized history usually improves usability.
- Do not force existing agents such as Codex CLI through the same prompt-context mechanism as API-level models. Use them as delegated agents and preserve their native context management.
- Keep durable project facts in canonical specs, schemas, YAML data, and tool-readable services rather than relying on chat memory. Chat sessions should remember the conversation, not become a hidden project database.
- Treat tool results as refreshable where possible. For example, validation diagnostics and record lists should be re-read by tools on demand instead of preserving large stale snapshots in every prompt.
- Keep confirmations and writes outside session replay. A resumed session may show that a proposal existed, but it must not replay a previous confirmation token or apply an old side effect.

- Resolved workspace root and `masterdata/` parent directory name for workspace scoping.
- Provider profile metadata from [AI provider configuration model](ai-provider-configuration-model.md).

- Tool results and diagnostics emitted by [AI assistant service](../component/ai-assistant-service.md).

## Writes

- Workspace-scoped AI session metadata and prompt-context files under `os.UserConfigDir()`.
- Compacted summary messages and compaction records.
- Retained run summaries, safe diagnostics, and UI-renderable tool event summaries.
- No canonical master data records, schemas, generation metadata, export settings, or raw credentials.
