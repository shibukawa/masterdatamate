# Template generate sample

This sample workspace demonstrates how MasterDataMate can combine runtime data exports with Pongo2 template-based code generation.

The domain is a small weather observation service:

- Runtime data: regions, weather stations, metrics, and providers are exported as JSON and loaded by the application at runtime.
- Generated sources: SQL DDL, Go constants/types, Go error definitions, and ingestion skeleton code are generated from table schemas and master data.
- Error messages: `error_message` is ordinary master data for code generation, but it is not exported as runtime JSON.

## Export Runtime Data

```bash
masterdatamate export \
  --workspace examples/templates \
  --format json \
  --output /tmp/mdd-json \
  --mkdirs \
  --force-overwrite
```

Expected runtime data files include:

- `/tmp/mdd-json/region.json`
- `/tmp/mdd-json/weather_station.json`
- `/tmp/mdd-json/weather_metric.json`
- `/tmp/mdd-json/weather_provider.json`

The application can read these JSON files at startup or deploy time to decide which regions, stations, providers, and metrics are active without recompiling the binary.
`error_message.json` is intentionally absent because error messages are compile-time inputs for generated Go constants and typed errors.

## Generate Sources

```bash
masterdatamate generate \
  --workspace examples/templates \
  --output-root /tmp/mdd-generated \
  --force-overwrite
```

Expected generated files include:

- `/tmp/mdd-generated/sql/schema.sql`
- `/tmp/mdd-generated/constants/types.go`
- `/tmp/mdd-generated/constants/errors.go`
- `/tmp/mdd-generated/constants/ingest_stub.go`
- `/tmp/mdd-generated/constants/stations_*.go`

The intended flow is:

1. Load `region.json` and `weather_station.json` as runtime configuration.
2. Apply `sql/schema.sql` to create the database tables.
3. Use the generated Go types and error definitions from `constants/`.
4. Implement the API-specific weather fetcher behind the generated ingestion skeleton.

## Run The Demo App

The sample app in `app/cmd/weatherdemo` uses both sides of the sample:

- JSON files exported from master data are runtime configuration.
- SQL and Go files generated by Pongo2 are build-time inputs.

From this directory:

```bash
masterdatamate export \
  --format json \
  --output runtime-json \
  --mkdirs \
  --force-overwrite

masterdatamate generate \
  --force-overwrite

go run ./cmd/weatherdemo \
  --json-dir runtime-json \
  --schema generated/sql/schema.sql
```

Expected output:

```text
Weather demo complete: 3 region(s), 3 station(s), 3 metric(s), 9 observation(s) written using sample_open_weather.
```

The demo applies `generated/sql/schema.sql` to an in-memory SQLite database, inserts exported `region` and `weather_station` data, then calls the generated `constants.RecordDailyWeather` skeleton with a deterministic fake fetcher.

## Template Definitions

`masterdata/generate_definitions.yaml` contains enabled template generation definitions. Its default `output_root` is `generated`, so normal project generation can run without passing an output path:

```bash
masterdatamate generate \
  --workspace examples/templates \
  --force-overwrite
```

- `sql_weather_schema`: project-level SQL DDL generation.
- `go_weather_types`: project-level Go struct generation.
- `go_error_definitions`: table-level Go constants and typed errors from `error_message`.
- `go_station_registry`: group-level station registry files grouped by `region_id`.
- `go_ingest_stub`: project-level ingestion skeleton.
