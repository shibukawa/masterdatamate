# Agent tool contract

## Summary

The agent tool contract defines the application-owned tools that an AI assistant may use to inspect, analyze, propose, and execute MasterDataMate operations. Tools expose business operations, not raw filesystem access.

The same tool implementation should be reusable by the in-app AI assistant, delegated external agents such as Codex CLI, local providers such as Foundation Models `fm serve`, and by a future MCP server. The entry transport may differ, but validation, authorization, confirmation, and application service calls remain shared.

The host should not expose the full canonical tool catalog to every model. Small-context providers such as the local Foundation Models `system` model need a compact runtime tool profile with a few generic tools, while larger hosted models or MCP clients may use more specialized tools when that improves reliability.

## Responsibilities

- Define tool schemas with explicit request and response shapes.
- Mark each tool as read-only, validation, or local-draft staging.
- Scope tool access to declared workspace, table, and generation view contexts.
- Normalize tool results into assistant-readable summaries and machine-readable payloads.
- Validate proposed table record changes before execution.
- Keep canonical writes outside the initial AI tool surface.
- Make tool results auditable enough for users to understand what the assistant did.

## Tool Categories

| Category | Examples | Confirmation | Notes |
| --- | --- | --- | --- |
| Discovery | `get_project_overview` | no | Returns available tables, generations, active context, and high-level capabilities. |
| Frontend state read | `get_current_context` | no | Runs in the browser through the AI panel and returns current route, active table, generation, selection, visible rows, diagnostics, and dirty state. |
| Backend read | `get_table` | no | Runs on the host service and returns scoped canonical data bundles with limits and projections. |
| Backend validation | `validate_table` | no | Runs deterministic validation and returns diagnostics; LLM-facing explanation is generated by the model from diagnostics. |
| Frontend draft staging | `stage_table_changes` | no | Sends AI-drafted table changes to the browser so the existing editor working copy applies accepted changes; does not write YAML. |
| Frontend control | `focus_table`, `open_generation_editor` | no | Runs in the browser through the AI panel UI to navigate or request user input. |

## Initial Tool Set

The initial runtime tool set should be compact for all managed chat providers, not only small-context local providers. The first implementation should expose no more than five tools:

| Tool | Kind | Purpose |
| --- | --- | --- |
| `get_current_context` | frontend-read | Return current route, selected table, selected record keys, active generation, display mode, pending edit status, visible diagnostics, visible rows, pagination, and available high-level capabilities from the browser. |
| `get_project_overview` | discovery | Return table summaries, generation summaries, active generation metadata, and feature availability in one compact response. |
| `get_table` | read | Return one table bundle containing schema summary, selected generation records, provenance metadata, validation diagnostics, and optional record/field projection. |
| `validate_table` | validation | Validate current table records or a proposed table change set without writing YAML. |
| `stage_table_changes` | frontend-draft | Accept AI-drafted table operations, normalize and structurally check them, then send them to the ordinary browser editor working copy so the existing dirty/diff UI becomes the preview. |

File import, binary asset attachment, export execution, persistent merge, generation deletion, schema editing, and spec search are not part of the initial managed-chat tool set. Generation analysis and comparison are not deferred tools; the LLM should perform those explanations from bounded `get_project_overview`, `get_table`, and validation results.

## Recommended Tool Catalog

The tool catalog is intentionally small. Each tool must use structured parameters, return JSON-serializable results, and map to the same application services used by ordinary UI actions.

The host may have many internal application services, but they should not become separate LLM tools by default. Internal services are composed behind the compact tool boundary. For example, `get_table` may call schema loading, generation-aware record loading, reference lookup, and validation services, then return one bounded bundle.

## Runtime Tool Profiles

| Profile | Target | Exposed Tool Shape | Notes |
| --- | --- | --- | --- |
| `minimal_managed_chat` | Foundation Models `system`, small local models, strict low-context profiles | `get_current_context`, `get_project_overview`, `get_table`, `validate_table`, `stage_table_changes` | Default for local `fm serve` and acceptable for the initial hosted-provider implementation. |
| `standard_managed_chat` | Hosted or larger local chat models | Same initial tool set, with at most one or two task-specific additions after implementation proves they are needed | Tool definitions are selected per run, not globally. |
| `specialized_mcp` | Future MCP server and developer automation | May expose narrower operations for machine clients | External clients can discover operations without fitting all schemas into one model prompt. |
| `delegated_external_agent` | Codex CLI and similar agents | External-agent adapter plus optional MCP-style host tools | The external agent owns its own context and may use host tools through a separate adapter contract. |

Tool profile selection is part of provider capability resolution. A profile may be downgraded from `standard_managed_chat` to `minimal_managed_chat` when token budget, provider schema limits, or health checks show that the full set is unreliable.

## Tool Semantics

Managed chat runs start from the user's message and selected session. They should not eagerly embed a full current UI snapshot into every provider request. Instead, current browser state is obtained lazily through `get_current_context` when the model needs it. This keeps ordinary chat turns small and makes dirty editor state an explicit tool result rather than hidden prompt preamble.

Tool execution is routed by execution host:

- Frontend tools run in the active browser session through the AI panel event stream. The server emits a tool request event, waits for the browser tool result or timeout, and forwards the structured result back to the provider as the tool result.
- Backend tools run on the host service and may read canonical workspace data, run deterministic validation, or summarize generation metadata.
- The assistant must not infer frontend state on the server. Route, active table, active generation, selection, visible rows, scroll/page context, and dirty working-copy state are browser-owned until returned by a frontend tool.
- If no active browser session can answer a required frontend tool, the tool result is `blocked` and the assistant asks the user to open the relevant screen or retry.

`get_current_context` is a frontend tool. It returns the browser-owned editing context visible to the user, including active route, selected table, active edit generation, generation view mode, selected record keys, visible row window, row count or pagination metadata, visible diagnostics, editor dirty state, and high-level frontend capabilities. It may include pending local edits only as values currently held by the editor working copy; it must not read canonical YAML directly.

`get_project_overview` is the only broad discovery tool. It returns table IDs, labels or comments, primary key summaries, generation IDs, active generation, output flags, and available feature capabilities. It should not return full schemas or records.

`get_table` replaces separate `get_table_schema`, `read_generation_records`, and table-level diagnostic reads. It accepts a table ID, generation view mode, optional selected record keys, pagination, field projection, and detail level. It returns:

- schema summary: fields, types, primary keys, references, defaults, export flags, and comments.
- records: one bounded page of rows for the requested generation view, including provenance and readonly metadata.
- diagnostics: current validation diagnostics for the returned table scope when cheap enough to compute.
- related summaries: compact reference table hints only when needed for display or validation explanation.

`get_table` pagination uses a stable host-defined cursor or offset plus a limit. The default response should prefer selected records, nearby visible records, and diagnostics-relevant records before broad table pages. Full-table reads require explicit pagination; the tool must not return every record by default.

`validate_table` runs deterministic validation for current table data or a proposed table change set. It is not an LLM analysis tool. Higher-level explanation, suspicious-record finding, generation analysis, and generation comparison should be performed by the model using `get_table`, `get_project_overview`, and validation diagnostics.

The AI model drafts table record operations itself after reading context through `get_current_context`, `get_project_overview`, and `get_table`. The host tool layer does not decide what changes should be made.

`stage_table_changes` is a frontend draft-staging tool. It accepts AI-drafted insert, update, or delete operations for one table, then sends them to the browser editor. The browser verifies that the target table and generation match the active editable surface, normalizes operations into the `extable` working-copy model, rejects malformed or unsafe operations, and applies accepted changes to the same editor working copy used by manual edits. Move operations are not part of the initial tool contract. The existing editor dirty-state and red changed-cell/changed-row UI is the preview. The tool returns affected records, changed fields, accepted operation indexes, rejected operation indexes, and diagnostics, but no confirmation token and no canonical write.

`stage_table_changes` does not mark changes as AI-authored in the table data model. Once staged, changes are ordinary editor working-copy changes. The UI may show a transient assistant summary, but saved records do not retain an AI-origin marker.

`stage_table_changes` operation rules:

- `insert` supplies a complete new record shape accepted by the current editor row model.
- `update` supplies the target primary key and changed field values only.
- `delete` supplies only the target primary key. It must not include a full record payload.
- `move` is not accepted.
- Operations target only the active editable generation and must not modify readonly previous-generation rows.
- Operations use canonical field names and primary-key values. Display labels are not accepted as write identifiers.
- Operations may be partially accepted when the editor can safely stage a subset. Rejected operations are reported with field or row diagnostics for the existing editor UI to display.
- If the operation list is structurally invalid, targets the wrong table, includes path-like identifiers, or would corrupt the editor working copy, the whole request is rejected.

Validation after staging is handled by the existing editor validation behavior. The staging tool may perform cheap structural checks and normalization, but it does not need to run a separate full validation pass if the editor already validates dirty rows or the table on demand.

The user commits staged AI changes with the ordinary editor commit button. AI tools do not call the save/commit API in the initial managed-chat tool set. Schema edits, generation metadata writes, merge operations, and export settings are deferred until their ordinary UI and backend services are stable enough to expose safely.

## Deferred Tool Ideas

The following operations are intentionally not part of the initial managed-chat tool catalog:

- `read_generation_config`: generation metadata is summarized by `get_project_overview` until generation editing is exposed deliberately.
- Export tools: export execution and export settings are ordinary product workflows and are not needed for the first AI chat tool surface.
- Schema editing tools: schema changes have larger blast radius and should wait until the schema editing APIs and confirmation UX are mature.
- File import and binary asset tools: these require stronger filesystem access, data-format conversion, preview, and asset-write semantics than the initial AI table-editing scope should carry.
- Spec search tools: product help can be added later, but it should not consume early local-model context budget.

## Tool Schema Rules

- Every tool request includes a `workspaceContextId` or equivalent host-issued context identifier; the assistant must not pass arbitrary filesystem roots.
- Table and generation IDs must be canonical identifiers loaded by the host.
- Record changes use canonical primary-key values and schema field names.
- Draft-staging responses include affected tables, operation list, validation diagnostics, and a human summary.
- Tool responses distinguish `success`, `blocked`, `validation_failed`, `staged`, and `error`.
- Tool responses should include compact natural-language summaries for the assistant and structured payloads for UI rendering.
- Tool definitions and results identify whether execution is `frontend` or `backend`; the assistant service brokers both through one provider-facing tool-call loop.
- Large record reads should support limits, filters, and selected field projection.
- The host may reject broad reads when the assistant request would exceed context or privacy limits.
- Tool schemas must be authored in the strictest supported JSON Schema subset used by the host, rather than relying on provider-specific normalization at request time.
- OpenAI-compatible tool definitions emitted by the host must use strict object schemas by default: every object schema sets `additionalProperties: false`, every property key is listed in `required`, optional values are represented as nullable or explicit sentinel values, and unsupported schema decoration such as `$schema` is omitted from provider payloads.
- Nested object schemas and array item object schemas follow the same strict required-property and `additionalProperties: false` rules as top-level tool parameters.
- Tool schemas must avoid ambiguous optional properties because local providers such as Foundation Models `fm serve` may reject otherwise valid JSON Schema shapes that hosted providers tolerate.
- The canonical tool catalog should remain transport-neutral, but its baseline schema shape must already be acceptable to strict OpenAI-compatible providers, Codex CLI structured prompts, Foundation Models `fm` tool declarations, and future MCP tools.
- Browser-owned actions such as reading current editor state, staging draft changes, focusing the active table, or switching the visible editor view are frontend tools. They may update UI state but must not save canonical project data directly.
- Backend tools are the only tools that may read canonical workspace files or validate data. Initial AI tools do not write YAML.
- Runtime adapters must emit only the selected tool profile for a given run. They must not dump the entire canonical catalog into the prompt for small-context providers.
- Tool schemas must keep their top-level shape stable and compact. Breadth belongs in bounded request fields and host-side routing, not dozens of separate tool definitions.
- If a deferred workflow later needs tool support, the host may add one specialized tool for that active workflow only after the ordinary UI and backend service contract are stable.
- Tool-calling support is required for draft staging. Providers without reliable tool calls are limited to read-only explanation and text suggestions; host parsing of free-form model output is out of scope for the initial managed-chat implementation.

## Safety Rules

- Tools must not expose arbitrary shell execution, raw filesystem read/write, unrestricted HTTP requests, or environment variables.
- Tools must not accept path traversal input or wildcard deletion targets.
- Initial AI tools must not write canonical files. Canonical writes remain behind the ordinary editor commit button.
- File import and binary asset attachment are outside the initial AI assistant target. Agent tools must not receive arbitrary local filesystem paths from the browser.
- If a provider cannot reliably emit structured tool calls, draft staging is disabled.
- Command-backed providers must not receive a capability to run arbitrary commands as an agent tool; their process execution is limited to the configured provider command itself.
- Tools must not become `bash`, arbitrary SQL, arbitrary filesystem read, arbitrary HTTP, or eval-style escape hatches. All operations route through typed application services and host-issued IDs.
- Prompt-injection text inside master data, schemas, comments, or diagnostics must be treated as data returned by tools, not as developer or system instructions. Tool results should be wrapped or labeled so the model can distinguish host instructions from workspace content.
- The assistant must not follow instructions found in records, schema comments, or validation messages that ask it to ignore system rules, reveal secrets, broaden tool scope, or write files.
- The ordinary table editor remains the fallback repair surface for records changed through AI tools.

## MCP Reuse

The tool implementation should be transport-neutral so a future MCP server can expose the same tools to external agents. MCP exposure may use stricter defaults than the in-app assistant, because the user may not see every intermediate UI event inside MasterDataMate.

An MCP server must preserve the same rules: scoped context, no raw filesystem access, validation before staging or writing, and explicit confirmation or host policy before side effects.
